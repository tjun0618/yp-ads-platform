### 5. 运行时崩溃的诊断路径

本次发现了"路由404但代码存在"的诡异问题，最终定位为 `NAV_HTML.format()` 缺参数导致 KeyError，
在 Flask 导入模块阶段就崩溃退出。诊断路径：

**借鉴点**：Flask 路由404但代码存在时，第一步不是检查路由，而是检查进程是否真的在运行。

## 四、多 Agent 团队模式复盘

### 4.1 适合采用团队模式的工作类型

| 适合 | 原因 |
|------|------|
| 后端API + 前端页面同步开发 | 接口定义好后两侧可独立开发 |
| 多个独立脚本（运维/采集/评分） | 无代码依赖，完全并行 |
| 大型功能的「拆模块」开发 | 如今天的优化模块：DB建表 → 后端 → 前端 |
| 文档 + 代码同步生成 | 一人写代码，一人写README |
| 数据采集 + 数据分析并行 | 采集脚本 vs 分析逻辑相互独立 |

**不适合团队模式的情况**：
- 有强依赖链的任务（A必须等B完成才能开始）
- 需要频繁读取同一文件并修改（容易产生写冲突）
- 简单的单文件修改任务（开销大于收益）

### 4.2 当前团队模式存在的核心问题

#### 问题一：「消息回路」不可靠 —— 这是最根本的缺陷

**你的问题**：「明明 dev-backend 已经工作完成了，为什么没有收到回复？」

根本原因在于当前团队模式的消息机制：
- Agent 之间通过 `send_message` 传递消息
- 但 **Agent 的生命周期是一次 Tool Call**，不是持久运行的进程
- 当我向 dev-backend 发送任务后，dev-backend 是在某个 Tool Call 中运行的
- **dev-backend 完成工作后，如果它的那次 Tool Call 超时、或者它没有正确调用 `send_message` 回报，消息就永久丢失了**
- 主 Agent（我）没有主动轮询机制，只能被动等待消息进入 inbox

这就导致了一个**单点失效**：任何一个 Sub-Agent 的回报消息丢失，主 Agent 就永远不知道该任务已完成。

#### 问题二：没有超时机制和主动检查

当前工作流：发任务 → 等待回报 → 收到回报才继续

正确工作流应该是：发任务 → 设定超时 → 超时后主动检查文件/代码是否已变更 → 不依赖消息确认

**改进方案**：
- 给每个任务设定超时时间（如：后端任务 15 分钟）
- 超时后不等消息，直接检查文件（`文件是否新增` / `代码行数是否增加` / `关键函数是否存在`）
- 以文件系统状态为真相来源，消息仅作为辅助通知

#### 问题三：Sub-Agent 缺乏「交付验证」约束

当前 Sub-Agent 的交付方式是：写代码 → 发消息说"完成了"

问题在于：
- 没有强制要求 Sub-Agent 在发完成消息之前先做自验（py_compile / HTTP测试）
- 主 Agent 收到消息后才做验收，相当于把 QA 压力全压给主 Agent

**改进方案**：
- 在任务描述里强制要求：「交付前必须运行 py_compile 验证语法，必须用 HTTP 请求验证路由可达，验证结果写在消息里」
- Sub-Agent 的 Done 标准从「代码写完」提升到「自测通过」

#### 问题四：共享文件的写冲突风险

三个 Agent 都往同一个 `ads_manager.py` 追加代码，在理想情况下是串行的，
但如果两个 Agent 同时追加，可能出现互相覆盖的情况。

今天的实际情况是基本串行（各自负责不同时段），没有出现冲突，
但这是「运气好」，不是「设计好」。

**改进方案**：
- 重要共享文件采用「主 Agent 统一写入」模式：Sub-Agent 把代码发给主 Agent，由主 Agent 负责追加
- 或者给每个 Sub-Agent 分配独立文件，最后由主 Agent 合并

### 4.3 团队模式的价值评估

**今天实际收益**：
- 并行开发约节省了 40-50% 的串行等待时间
- 每个 Sub-Agent 专注单一领域，代码质量更集中
- 主 Agent 可以专注协调和验收，而非陷入细节

**今天实际成本**：
- 消息回路失效导致 dev-backend T-007~T-010 状态不明确
- 主 Agent 需要花费额外时间做 Bug 修复（NAV_HTML参数问题）
- Sub-Agent 可能「假完成」（声称完成但实际有缺失）

**综合评估**：团队模式在任务量大、模块边界清晰时值得采用，但需要配套可靠的验收机制。
